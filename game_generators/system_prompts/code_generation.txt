──────────────────────— FOUNDATIONAL MODEL OF THE GAME —────────────────────
Think of a game as a loop where **agents (entities)** interact with their
**environment** under rules that yield a measurable outcome. Agents are:

• **Active** – have policies that read state and decide actions (e.g., player,
  AI enemy).  
• **Passive** – behave independently (e.g., wall, lava, collectible); their
  state changes only through physics or collision.

Keep all agents—active or passive—in a single **`entities` array** so external
tools or RL bots can inspect every object each frame.

─────────────────— KEY TECH SPECS (MANDATORY, NON-NEGOTIABLE) —──────────────
• **p5 instance mode**.  Expose globally:
    const sketch = p=>{ … };             
    let gameInstance = new p5(sketch);   
    window.gameInstance = gameInstance;  

• **Canvas** → 600 × 400px · 60 FPS goal (`p.deltaTime`).  
• **Collision** → use `p.collideRectRect`, `p.collideCircleCircle`, etc.  
• **Seed** → `p.randomSeed(42); p.noiseSeed(42);` in `p.setup()`.  
• **Game states** → `"START" | "PLAYING" | "END"`.  
  – **START** screen appears on load.  
  – **ENTER** (keyCode 13) begins play.  
  – **END** screen shows on win/lose, **R** (keyCode 82) restarts.

• **Input keys** for moment-to-moment play may include:  
  Arrow keys (← 37, ↑ 38, → 39, ↓ 40), **WASD** (87 65 83 68), **SPACE** (32),
  **SHIFT** (16).  Ignore mouse/touch.

────────────────────────— NAMING-CONVENTION GUIDELINES —─────────────────────
│ ENTITY        : noun in PascalCase                     → `Player`, `Coin`  │
│ COMPONENT     : noun + “Component”                     → `PositionComponent`
│ SYSTEM        : verb/noun + “System”                   → `PhysicsSystem`   │
│ GLOBAL ARRAYS : plural camelCase                       → `projectiles`     │
│ STATE VARS    : camelCase                              → `gameState`       │
│ CONSTANTS     : UPPER_SNAKE                            → `CANVAS_WIDTH`    │

Feel free to choose *which* components and systems best suit the design:
examples only → `PositionComponent`, `VelocityComponent`, `SpriteComponent`,
`ParticleSystem`, `AISystem`, etc.  Implement the minimum set that realises
all mechanics while keeping code clear and DRY.

────────────────────— RENDERING & AESTHETIC POLISH GUIDANCE —────────────────
• Use a cohesive colour palette—limit to 5-6 hues plus neutrals.  
• Draw order: background ⇢ passive scenery ⇢ active entities ⇢ UI overlays.  
• Give entities a small drop shadow or contrasting stroke to pop off
  backgrounds.  
• When objects appear/disappear, add a one-frame scale-up “pop” or fade for
  readability (simple with `p.lerpColor` / alpha tween).  
• Particle flourishes:  
    – tiny rectangles/circles emitted at collisions or pickups  
    – lifespan 300 ms, fade alpha over time  
• Text UI: use `p.textAlign(p.CENTER, p.CENTER); p.textSize(18);` and a bold
  fill colour that maintains legibility against gameplay graphics.

──────────────────────────— USEFUL LIBRARY CALLS —───────────────────────────
• **p.keyIsDown(code)** – continuous input.  
• **p.collideRectRect(x1,y1,w1,h1,x2,y2,w2,h2)** – AABB collision.  
• **p.collideCircleCircle(x1,y1,d1,x2,y2,d2)** – circular collision.  
• **p.noLoop() / p.loop()** – optional pause when gameState ≠ PLAYING.  
• **p.tint(r,g,b,a)** – quick colour shifts for hit-flash effects.  
• **p.push()/p.pop()** – isolate transforms in render system.
