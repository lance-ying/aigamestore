-	No external images or fonts

-	Write modular structured code with proper ES6 imports and exports resulting in an error-free game following the code architecture and state management instructions
-	Maintain p.logs as write only (see p5_usage)
-	Implement automated game testing functions to validate different aspects of the game (see automated_game_testing section)

<game_design_process>
Before implementing the code for the game concept, plan the complete player-experience-centric game design in natural language. DO NOT output this game design plan.
- Core loop: Think about an engaging, learnable, and intuitive core gameplay loop of the game
- Entities: Define the entities including the player, characters, and interactive objects with their objectives, behaviors, and interactions with other entities
- Player abilities: Define the player actions and abilities and how they can be used to achieve the objectives
- Environment: Carefully choose the setting, viewpoint, and objects that have an intent and purpose in the game  
- Rules: simple small set of rules that can lead to emergent complexity
- Objectives:
	- Win condition: Explicit achievable goal for the player
	- Losing conditions: Choose the losing conditions that are fair and consistent
	- Rewards: Plan the rewards for all objectives
	- Subgoals: Add side goals if needed to motivate the player to achieve the final goal
- Progression: Plan a smooth progression ramp starting from easy to challenging yet achievable at the highest difficulty. Add interesting entities, mechanics, and challenges to keep the player engaged and motivated.
- Visual design: The visual appearance of the game environment and entities
</game_design_process>

	```javascript

		... // other game state variables which are initialized for the game and would be useful for automated testing
	};
	```

<p5_usage>
- Use p5.js in instance mode and store the p5 instance in a variable called `gameInstance`. Example:
```javascript
	const p5 = window.p5
let gameInstance = new p5(p => {
	// Initialize the logs. Important: Do not reset the logs at any point in the code! logs are considered write-only and never re-assigned!
	p.logs = {game_info:[], inputs:[], player_info:[]}

	...
});
// Expose globally
window.gameInstance = gameInstance;
```
- p.logs is write-only. Don't reset it. Keep track of all the logs since the start of the p5.js game instance. This information is logged for both HUMAN and TEST modes.
- Variables to store in the p.logs during gameplay:
	- In "game_info": Information about the game when there are changes in the gamePhase, controlMode, etc.
		- "game_status": The game phase status of the game accessible via `gameState.gamePhase` where `gameState` is accessible via `window.getGameState()`
		- "data": Additional data specific to the game state. For example, the player's score when game is over. Leave empty if not applicable.
		- "framecount" and "timestamp": The framecount and timestamp of the event accessed using `p.frameCount` and `Date.now()` respectively
	- In "inputs": Store the control inputs when they are triggered by the player.
		- "input_type": The input event type (keyPressed, keyReleased, etc.)
		- "data": Additional data specific to the input type. Store `key` and `keyCode` if the input is a key
		- "framecount" and "timestamp"
	- In "player_info": Information about the player character when there are changes in the player's state (accessed via `gameState.player` where `gameState` is accessible via `window.getGameState()`)
		- "screen_x", "screen_y": The x and y position of the player on the screen
		- "game_x", "game_y": The x and y position of the player in the game world
		- "framecount" and "timestamp"

- Useful Functions:
	- Use `p` to access p5.js functions to avoid ReferenceError: p is not defined
	- p5: keyIsDown, keyPressed, noLoop/loop, tint, push/pop, textAlign, textSize, text  
	- p5.collide2D function name format (order–matters, 'collideCircleRect' is not available but 'collideRectCircle' is available):
		- collidePoint{Point|Circle|Ellipse|Rect|Line|Arc|Poly|Triangle}
		- collideRect{Rect|Circle|Poly}
		- collideCircle{Circle|Poly}
		- collideLine{Line|Circle|Rect|Poly}

<automated_game_testing>
- Write test functions to validate the different aspects of the game based on the game description, controls, and the game code. These tests will be used to evaluate the functionality of the game by a human evaluator.
	- TEST_1: getStickyKeysAction(gameState): Demonstrate that the game compiles and runs without errors, played using sticky keys (continuous action sequence of a random length between 4 and 16 changing actions every now and then).
	- TEST_2: getTestWinAction(gameState): Demonstrate that a player can win the game. Implement one of the best possible strategies to win the game.
	- Choose up to 5 more tests
	- For each test, provide a test description, a strategy description, and the expected outcome based on the testing strategy
- Responsibilities of the evaluator and the test function in the automated testing flow when controlMode is not HUMAN:
	- The evaluator (using game phase controls):
		- select and starts the automated testing for the selected test mode by pressing ENTER
		- If game phase is GAME_OVER_WIN or GAME_OVER_LOSE, evaluator is responsible for restarting the game using the R and ENTER keys
		- If game phase is PLAYING, the evaluator cannot use game controls. The test function is responsible for playing the game.
	- The test function (using game controls):
		- Plays the game as soon as the game phase is PLAYING based on the test function outputting game control keys
		- Do not change the game phase or use game phase controls. 
- The implementation of the test function will be responsible for the following:
	- Input: gameState object
	- Output: Return control key code or null to the input handling overriding the human control key inputs to play the game
	- You can return the same control key to mimic pressing down on a key, shifting between keys to perform action sequences such as jumping and moving right at the same time, and having delays between actions
	- The parameters and logic must be a reasonable implementation to test the desired aspect of the game following the strategy description to achieve the expected outcome
	- Prevent player from getting stuck by tracking position history; if stuck, inject random actions to resume progress

	- Expose globally as window.game_testing_controller
